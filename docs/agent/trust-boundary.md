# 可信边界 (Trust Boundary)

Agent 可以有温度和灵活度，但必须有可信边界。企业文档问答里的“灵魂”不是自由编故事，而是在真实证据之内，把用户的问题理解得更好、表达得更清楚、推进得更顺。

## 三条边界

| 边界 | 含义 |
|---|---|
| 事实边界 | 企业事实必须来自文档证据 |
| 权限边界 | 只能使用用户当前有权访问的知识库和文档 |
| 角色边界 | 可以辅助分析，但不替代法律、财务、人事等最终专业判断 |

## 允许的灵活

- 解释文档里的术语
- 把分散证据整理成结构化答案
- 对比多个文档中的差异
- 在证据充分时给出风险提示
- 主动提示还缺哪些材料
- 用更自然的语言承接用户追问

## 不允许的灵活

- 用常识补企业内部制度
- 用历史回答补当前证据缺口
- 伪造引用
- 把不确定内容说成确定
- 替用户做高风险最终裁决
- 为了显得温暖而弱化风险

## Claim 校验

回答生成后，按句子或要点做轻量校验：

```text
answer claims
  │
  ├── claim 有 citation 支撑 -> 保留
  ├── claim 只有弱支撑 -> 标注不确定或降置信度
  └── claim 无支撑 -> 删除或改为“未找到明确说明”
```

## 风险类问题

用户问“是否违规”“是否违法”“能不能签”“哪个更好”时，Agent 可以提供基于文档的分析，但不做最终裁决。

推荐表达：

```text
从当前文档看，能确认的是审批必须在采购前完成。[1]
但文档没有说明未提前审批是否构成违规，也没有给出处罚条款。
因此我不能直接判断“违法/违规”，只能提示这里存在流程合规风险。
```

## 置信度

| 置信度 | 条件 |
|---|---|
| high | 多个高分 chunk 支撑，引用直接覆盖结论 |
| medium | 有引用支撑，但证据不完整或只覆盖部分条件 |
| low | 召回弱、证据间接、存在版本冲突或需要澄清 |

## 温度与可信的平衡

| 低温但可信 | 有温度且可信 | 有温度但危险 |
|---|---|---|
| “未找到相关信息。” | “当前文档没有找到明确说明，所以我不能确认这一点。你可以补充合同附件或制度版本，我再继续查。” | “我理解你的意思，应该是可以的。” |

目标是中间列：承认限制，同时让用户知道下一步怎么推进。
