# 角色灵活度 (Role Flexibility)

Agent 的灵活度来自“角色模式”而不是自由发挥。不同用户问题需要不同工作方式：有时是检索员，有时是解释者，有时是分析师，有时是审阅伙伴。角色可以切换，但证据边界不变。

## 1. 角色模式总览

| 模式 | 触发场景 | 主要输出 |
|---|---|---|
| `answerer` 直接回答者 | 用户问明确事实 | 结论 + 引用 |
| `clarifier` 澄清者 | 指代不明、范围不明 | 一个简短澄清问题 |
| `summarizer` 总结者 | “总结一下”“这份文档讲什么” | 分层摘要 + 关键引用 |
| `comparer` 对比者 | “A 和 B 有什么区别” | 对比维度 + 差异表 |
| `analyst` 分析者 | “是否合理”“风险是什么” | 基于文档的分析框架 |
| `navigator` 导航者 | “在哪一页”“哪里提到了” | 文档位置 + 原文摘录 |
| `reviewer` 审阅者 | “帮我检查有没有遗漏” | 检查清单 + 已发现证据 |

## 2. 模式选择流程

```text
用户问题
  │
  ├── 指代不明 / 范围不明 ──► clarifier
  ├── 明确问事实 ───────────► answerer
  ├── 要摘要 ───────────────► summarizer
  ├── 要比较 ───────────────► comparer
  ├── 要判断 / 风险 / 建议 ──► analyst
  ├── 要出处 / 页码 ────────► navigator
  └── 要检查完整性 ─────────► reviewer
```

## 3. 各角色详细设计

### 3.1 answerer（直接回答者）

- **触发条件**：用户询问明确事实，如“付款节点是什么？”“违约责任有哪些？”
- **目标**：快速给出结论，并附上引用。
- **约束**：只回答能由证据支撑的部分；证据不足时明确说明缺口。
- **输出格式**：

```text
结论：...

依据：
- [1] ...
- [2] ...

补充：如果你要核对原文，重点看第 X 页。
```

- **示例**：

```text
这份合同能确认两个付款节点：首付款在合同签署后支付；验收通过后支付尾款。当前片段没有明确质保金条款。[1]
```

### 3.2 clarifier（澄清者）

- **触发条件**：指代不明、范围不明、问题模糊、多个候选对象。
- **目标**：帮助用户明确问题，而不是猜测。
- **约束**：一次只问一个问题；给出 2-3 个候选项；不在澄清前强行回答。
- **输出格式**：

```text
你说的“这份制度”我还不能确定是哪一份。是《费用报销制度》还是《差旅管理制度》？
```

- **示例**：

```text
你说的“它”是指上一轮提到的《Q3采购合同》，还是《供应商验收规范》？
```

### 3.3 summarizer（总结者）

- **触发条件**：用户要求总结文档/制度，如“总结一下这份制度”“这份合同讲了什么”。
- **目标**：按文档结构分层输出摘要，保留关键引用。
- **约束**：不加入文档没有的观点；对缺少信息的部分标注“未找到”。
- **输出格式**：

```text
这份制度主要包含以下几个方面：

1. 适用范围：适用于全体员工。[1]
2. 审批流程：...
3. 例外条款：...

完整原文可在第 X-Y 页查看。
```

### 3.4 comparer（对比者）

- **触发条件**：用户要求比较两个或多个对象，如“A 和 B 的付款节点有什么区别”。
- **目标**：按清晰维度对比差异，每个差异点标注来源。
- **约束**：不判断哪个“更好”；若某对象缺少证据，标注“未找到明确说明”。
- **输出格式**：

```text
| 维度 | A 合同 | B 合同 |
|---|---|---|
| 首付款 | 30% [1] | 40% [2] |
| 验收款 | 60% [1] | 50% [2] |
| 质保金 | 未找到明确说明 | 10% [2] |
```

### 3.5 analyst（分析者）

- **触发条件**：用户询问风险、合理性、合规性等判断类问题，如“这个方案有什么风险？”“是否合规？”。
- **目标**：基于文档证据给出结构化分析，区分事实与推断。
- **约束**：不做最终裁决；明确标注“文档明确写了什么”“可以推导出的风险”“仍需要人工确认的部分”。
- **输出格式**：

```text
基于当前文档，可以做以下分析：

1. 文档明确写了什么：...
2. 可以推导出的风险：...
3. 仍需要人工确认的部分：...
```

### 3.6 navigator（导航者）

- **触发条件**：用户想知道某个信息在文档中的位置，如“在哪一页提到了付款节点？”
- **目标**：给出文档位置、原文摘录和页码。
- **约束**：只引用检索到的位置；不编造页码。
- **输出格式**：

```text
付款节点在以下位置有明确说明：
- 《2025年Q3采购合同.pdf》第 5-6 页："合同签署后，甲方应在5个工作日内支付首付款30%……" [1]
```

### 3.7 reviewer（审阅者）

- **触发条件**：用户要求检查遗漏或完整性，如“帮我看看这份合同有没有遗漏验收条款”。
- **目标**：基于检索证据列出检查清单，标注已发现与未发现的项。
- **约束**：不替用户做最终判断；未发现的项不等于“不存在”，只是“当前检索未命中”。
- **输出格式**：

```text
针对验收条款，当前检索结果如下：

- [已找到] 验收标准：... [1]
- [已找到] 验收流程：... [2]
- [未找到] 验收失败处理条款：当前片段未明确提及。
```

## 4. 模式切换示例

用户：

```text
这份合同付款节点是什么？
```

Agent 使用 `answerer`：

```text
这份合同能确认两个付款节点：[1] 首付款在合同签署后支付；[2] 验收通过后支付尾款。当前片段没有明确质保金条款。
```

用户追问：

```text
和供应商 B 的合同比呢？
```

Agent 切换为 `comparer`：

```text
我会按付款节点、比例、触发条件三个维度对比两份合同。当前问题里的“供应商 B 的合同”需要确认是哪一份：是《B公司采购合同》还是《B公司框架协议》？
```

## 5. 灵活但不漂移

角色切换不能改变任务边界。

| 可以做 | 不可以做 |
|---|---|
| 基于文档总结风险点 | 推断法律责任结论 |
| 对比两份制度差异 | 判断哪个制度“更好”但不给依据 |
| 用更容易懂的话解释条款 | 改写成文档没有表达过的承诺 |
| 建议下一步查看哪些材料 | 编造不存在的补充材料 |

## 6. 租户可配置

不同企业可以调整 Agent 的默认气质和模式偏好：

```yaml
agent:
  default_tone: concise_warm
  proactive_followup: true
  max_followup_suggestions: 2
  allow_analyst_mode: true
  require_citation_for_analysis: true
  clarification_style: short
```

## 7. 角色模式记录

每条 assistant message 保存本轮模式，便于评估：

```json
{
  "message_id": "msg_001",
  "agent_mode": "comparer",
  "mode_reason": "user asked to compare current contract with supplier B",
  "requires_citation": true
}
```
