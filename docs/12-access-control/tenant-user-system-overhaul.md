# DocuMind 租户与用户系统重构方案

> 状态：实施基线
> 更新日期：2026-07-21
> 适用范围：DocuMind 独立认证模式及可选 Portal SSO 模式

## 1. 文档目的

本文定义 DocuMind 租户与用户系统的目标架构、角色边界、页面信息架构、数据模型、邀请与删除流程、租户隔离要求、迁移策略和验收标准。

DocuMind 不是单租户知识库工具，而是一套多租户企业文档智能问答平台。平台需要同时解决三类使用者的问题：

1. 超级管理员负责平台运行和租户生命周期。
2. 租户管理员负责本租户成员、知识库、文档和权限。
3. 普通用户只在授权知识库范围内问答并管理自己的账号信息。

核心原则是：平台管理权、租户管理权和业务数据访问权必须显式分离，任何身份都不能因为页面入口或角色名称混乱而获得额外数据访问能力。

## 2. 产品模型

```text
DocuMind 平台
├── 平台域
│   └── 超级管理员
│       ├── 平台健康与全局服务
│       ├── 租户生命周期
│       ├── 全局账号排障
│       └── 全量审计
└── 租户域
    ├── 租户 A
    │   ├── 租户管理员
    │   ├── 普通用户
    │   ├── 知识库
    │   ├── 文档、切片与索引
    │   └── 对话、引用与反馈
    └── 租户 B
        └── 与租户 A 完全隔离
```

系统最终只保留三类对用户可见的身份：

| 显示名称 | 目标代码 | 作用域 | 说明 |
| --- | --- | --- | --- |
| 超级管理员 | `super_admin` | 全平台 | 平台身份，不等价于任意租户管理员 |
| 租户管理员 | `tenant_admin` | 单租户 | 管理当前租户的成员与知识资产 |
| 普通用户 | `end_user` | 单租户内授权范围 | 仅访问授权知识库并进行问答 |

`viewer` 可以作为后续权限模板存在，但不作为本轮核心身份。`enterprise_admin`、`team_admin`、`data_admin`、`tenant_owner`、`user`、`analyst` 等历史角色必须通过兼容映射迁移到目标角色，不再继续扩散到新接口和新页面。

如果后续确实需要“用户管理员”，应将其设计为租户级权限模板 `member_admin`，只拥有成员管理能力，不形成第四种平台身份。

## 3. 现状结论

### 3.1 已有基础

当前系统已经具备以下多租户基础：

- `app_user`、`tenant`、`tenant_member` 组成全局账号与租户成员关系。
- `knowledge_base`、`documents`、`chunks`、`chunk_embeddings` 等核心资源均保存 `tenant_id`。
- `knowledge_base_acl` 支持按角色或用户授予 `read`、`write`、`manage`。
- 登录后由服务端解析当前用户、当前租户、角色、权限和授权知识库。
- PostgreSQL 查询、Elasticsearch dense/BM25 检索、文档预览和对话均有租户过滤。
- 租户邀请已支持 token hash、接受、撤销、重发和过期状态。
- 真实服务器 `tenant-integrity` 检查中，文档、切片、向量和来源锚点的租户错配均为 0。

### 3.2 当前问题

#### 角色语义混乱

前后端同时存在多套历史角色名称。部分角色具有相同能力但不同默认入口，前端又使用 `roles[0]` 决定跳转，容易出现角色顺序影响页面入口的问题。

#### 平台身份与租户身份混用

当前超级管理员仍作为默认租户的 `tenant_member` 存在，并在默认租户知识库 ACL 中具有 `manage`。这会让平台运维身份天然拥有业务知识权限，不符合租户信任边界。

本轮先建立独立平台管理员关系作为超级管理员授权来源，移除超级管理员的默认知识库授权和租户后台入口；原有默认租户成员关系仅作为兼容登录上下文，后续可以在令牌支持无租户平台上下文后彻底移除。

#### 后台导航混合

当前 `/system`、`/admin`、`/tenant` 和多套 Sidebar 同时存在。超级管理员页面同时展示“平台运维”“当前租户”“知识库配置”，租户配置又被拆成多个一级菜单，导致用户无法快速判断自己正在管理平台还是租户。

#### 平台租户管理只读

当前超级管理员只能查看租户列表和详情，不能创建租户、邀请首位管理员、变更状态或删除租户。

#### 租户成员管理不完整

当前邀请已有基础闭环，但页面固定 7 天有效期，成员列表缺少角色变更、停用、恢复和移除能力，也没有“至少保留一位租户管理员”的领域规则。

#### 普通用户缺少账号入口

普通用户聊天侧栏左下角没有头像、个人资料、当前租户、租户切换和退出入口。

#### 数据库隔离仍可加固

当前隔离主要依靠应用查询条件和数据完整性巡检。部分表虽然同时保存 `tenant_id`、`kb_id`、`doc_id`，但数据库未用复合外键强制这些 ID 必须属于同一租户。

## 4. 角色与权限边界

### 4.1 超级管理员

超级管理员负责平台，而不是默认拥有全部租户业务数据。

允许能力：

- 查看平台健康、依赖状态、任务和全局统计。
- 创建租户并邀请首位租户管理员。
- 编辑租户基本信息、套餐、配额和状态。
- 停用、恢复、归档和发起删除租户。
- 查看全局账号及其租户成员关系。
- 管理全局模型服务、向量索引和运行配置。
- 查看全量审计日志。
- 撤销或重发租户激活邀请。

默认禁止：

- 直接查看租户文档原文。
- 以超级管理员身份发起跨租户问答。
- 静默进入租户后台。
- 绕过知识库 ACL 获取引用和预览。

未来如需平台协助租户，应引入显式、限时、可审计的“租户代管会话”。本轮不开放静默代管。

### 4.2 租户管理员

租户管理员只管理当前租户。

允许能力：

- 查看租户概览和使用情况。
- 创建、编辑、归档和删除知识库。
- 上传、移动、重解析、排除检索和删除文档。
- 查看解析、清洗、切片和向量化状态。
- 邀请普通用户或其他租户管理员。
- 设置邀请有效期和初始知识库授权。
- 修改成员角色、停用、恢复或移除成员。
- 配置知识库 ACL。
- 查看本租户问答日志和审计日志。
- 管理租户级知识库默认配置与模型绑定。
- 作为用户发起当前租户内问答。

硬性限制：

- 不能创建或授予 `super_admin`。
- 不能操作其他租户。
- 不能移除或降级最后一位有效租户管理员。
- 不能修改全局 Provider 密钥和平台级索引。

### 4.3 普通用户

普通用户仅在授权知识库范围内使用问答能力。

允许能力：

- 查看自己可访问的知识库。
- 在一个或多个授权知识库中创建对话。
- 查看、收藏、重命名和删除自己的会话。
- 查看答案引用和有权限的原文。
- 提交点赞、点踩、错误原因和修正建议。
- 修改姓名、头像等个人资料。
- 查看当前租户和角色。
- 在自己所属的多个租户之间切换。
- 退出当前会话。

禁止能力：

- 访问 `/system/*` 和 `/admin/*`。
- 查看成员、邀请、权限和审计。
- 上传、移动、重处理或删除文档。
- 查看未授权知识库名称、文档、切片、引用和缓存结果。

## 5. 页面信息架构

### 5.1 平台后台 `/system`

超级管理员登录默认进入 `/system`。侧栏只展示平台内容：

```text
平台
  平台概览
  租户管理
  全局账号

运行
  模型服务
  向量索引
  任务队列

安全
  全局审计
  系统设置
```

不得在平台侧栏混入“当前租户”“文档管理”“知识库配置”和“返回对话”。

#### 租户列表 `/system/tenants`

列表展示：

- 名称和 slug。
- 状态。
- 租户管理员和成员数量。
- 知识库、文档、切片数量。
- 套餐和配额。
- 最近活跃时间。
- 邀请状态。

页面操作：

- 新建租户。
- 查看详情。
- 编辑基本信息。
- 停用或恢复。
- 归档。
- 发起删除。

#### 新建租户

使用分组清晰的抽屉或向导：

1. 租户名称、slug、套餐。
2. 首位租户管理员邮箱与姓名。
3. 邀请有效期，默认 7 天，允许 1～30 天。
4. 创建 `pending` 租户和邀请。
5. 展示并复制一次性邀请链接。

邀请接受后租户变为 `active`。如果希望提前使用，也可以由超级管理员手动激活。

#### 删除租户

租户删除不是即时级联删除：

```text
active → suspended → archived → deletion_pending → purged
```

本轮提供 `deletion_pending`：

- 删除前要求输入租户 slug 确认。
- 停止该租户登录和新任务。
- 撤销未接受邀请。
- 保留数据以便恢复。
- 写入全局审计。

物理清理必须由后续独立后台任务完成，并覆盖 PostgreSQL、对象存储、Elasticsearch、Redis 与任务数据。

### 5.2 租户后台 `/admin`

租户管理员登录默认进入 `/admin`。建议侧栏：

```text
当前租户
  租户概览

知识资产
  知识库
  文档管理

成员与安全
  成员与邀请
  权限策略
  问答日志
  租户审计

租户设置
  基本信息
  模型绑定
  用量与配额
```

切割策略、检索参数和模型绑定不再全部占据一级导航：

- 知识库级切割和检索配置进入知识库详情。
- 文档切片进入文档详情。
- 租户默认模型绑定进入租户设置。
- 全局模型 Provider 仍只属于 `/system/models`。

#### 成员与邀请 `/admin/members`

页面分为：

- 成员。
- 待处理邀请。
- 角色说明。

邀请表单字段：

- 邮箱。
- 姓名，可选。
- 角色：租户管理员或普通用户。
- 初始知识库授权：全部、指定或暂不授权。
- 有效期：1、3、7、14、30 天。

成员行操作：

- 修改角色。
- 停用或恢复。
- 移除成员。
- 查看加入时间和最近活跃时间。

邀请行操作：

- 复制本次生成的链接。
- 重发并生成新 token。
- 撤销。
- 查看接受、撤销和过期状态。

### 5.3 普通用户工作台 `/chat`

普通用户不展示后台侧栏。聊天侧栏包含：

- 新会话。
- 收藏会话。
- 按时间分组的历史会话。
- 底部账号入口。

底部账号入口显示头像、姓名和当前租户。展开菜单包含：

- 个人资料。
- 当前租户和角色。
- 租户切换。
- 管理后台，仅租户管理员显示。
- 退出登录。

个人设置统一放在 `/account`，不为不同角色维护重复页面。

## 6. 数据模型

### 6.1 全局账号

`app_user` 只表达自然人或服务账号本身：

- 邮箱和姓名。
- 密码 hash 或 SSO subject。
- 账号状态。
- 最近活动租户。
- 头像和个人偏好。

### 6.2 平台管理员

新增独立 `platform_admin`：

```sql
platform_admin (
  user_id UUID PRIMARY KEY REFERENCES app_user(id),
  role TEXT NOT NULL CHECK (role = 'super_admin'),
  status TEXT NOT NULL,
  created_by UUID,
  created_at TIMESTAMPTZ,
  updated_at TIMESTAMPTZ
)
```

平台 API 的超级管理员判断以此表为权威来源。兼容期内可以继续读取历史 `tenant_member.roles`，但新授权只能写 `platform_admin`。

### 6.3 租户与成员

`tenant` 状态至少支持：

- `pending`
- `active`
- `suspended`
- `archived`
- `deletion_pending`

`tenant_member` 角色目标值：

- `tenant_admin`
- `end_user`

成员状态：

- `active`
- `suspended`
- `removed`

同一 `app_user` 可以属于多个租户，并在不同租户拥有不同角色。

### 6.4 邀请

复用 `tenant_invitation`，同时支持：

- 超级管理员为新租户邀请首位管理员。
- 租户管理员为当前租户邀请成员。

邀请只保存 `token_hash`，明文 token 只在创建或重发响应中出现一次。

状态：

- `pending`
- `accepted`
- `revoked`
- `expired`

接受邀请时：

1. 校验 token、状态、有效期和租户状态。
2. 已有账号复用 `app_user`，新账号创建并设置密码。
3. 创建或恢复 `tenant_member`。
4. 写入初始知识库 ACL。
5. 标记邀请 accepted。
6. 更新 `last_active_tenant`。
7. 写审计并签发绑定当前租户的 session。

## 7. 租户上下文与切换

登录流程：

```text
校验账号
→ 加载平台角色和全部有效租户成员关系
→ 只有平台角色：进入 /system
→ 单个租户：进入对应默认页面
→ 多个租户：优先 last_active_tenant，否则展示选择器
→ 签发绑定 active tenant 的令牌
```

切换租户必须调用后端接口重新签发令牌，不能只修改前端 `localStorage.tenantId`。

切换后必须清空：

- 当前知识库列表。
- 对话列表和消息缓存。
- 文档预览缓存。
- 任何带租户范围的 React state。

## 8. 租户隔离标准

### 8.1 身份层

- 当前租户来自经过验证的 session/JWT。
- API 不信任请求体中的 `tenant_id`、`roles` 和 `user_id`。
- 租户状态非 active 时拒绝普通业务访问。

### 8.2 PostgreSQL

- 所有业务表保存 `tenant_id`。
- 所有查询显式带 `tenant_id`。
- 为 `knowledge_base`、`documents`、`chunks`、`chunk_embeddings` 增加复合唯一键和复合外键。
- 文档、切片、向量的父子关系必须同时匹配 tenant、KB 和 document。
- 后续引入 RLS 时，通过事务内租户上下文执行，不允许使用宽泛的全表策略。

### 8.3 Elasticsearch

- 索引文档必须保存 keyword 类型的 `tenant_id` 和 `kb_id`。
- dense、BM25、删除、重建都必须带 tenant filter。
- 普通用户查询还必须带 `allowed_kb_ids`。
- 后续建议以 tenant id 作为 routing key。

### 8.4 对象存储

对象键必须使用：

```text
tenants/{tenant_id}/knowledge-bases/{kb_id}/documents/{doc_id}/...
```

预览链接必须由 DocuMind 按当前租户与知识库权限签发，不能直接暴露长期对象地址。

### 8.5 缓存与任务

- Redis key 必须包含 tenant id。
- 回答缓存必须包含 tenant id、KB 范围和文档版本。
- 解析、OCR、Embedding 任务消息必须携带 tenant id。
- Worker 更新状态时同时匹配 tenant id 和资源 id。

### 8.6 审计

必须记录：

- 租户创建、状态变更和删除请求。
- 邀请创建、重发、撤销和接受。
- 成员角色、状态和移除。
- 知识库授权变更。
- 文档删除和重处理。
- 平台代管或敏感配置变更。

## 9. API 契约

### 9.1 平台租户 API

```text
GET    /api/system/tenants
POST   /api/system/tenants
GET    /api/system/tenants/{tenant_id}
PATCH  /api/system/tenants/{tenant_id}
DELETE /api/system/tenants/{tenant_id}?confirm_slug=...
```

创建返回租户和首位管理员邀请。删除只进入 `deletion_pending`。

### 9.2 租户成员 API

```text
GET    /api/admin/members
PATCH  /api/admin/members/{user_id}
DELETE /api/admin/members/{user_id}
GET    /api/admin/invitations
POST   /api/admin/invitations
POST   /api/admin/invitations/{id}/resend
POST   /api/admin/invitations/{id}/revoke
```

### 9.3 当前账号与租户 API

```text
GET   /api/account/profile
PATCH /api/account/profile
GET   /api/account/tenants
POST  /api/account/switch-tenant
```

## 10. 领域规则与错误

所有缺失路径返回明确错误，不使用隐式兜底：

| 规则 | 错误码 |
| --- | --- |
| slug 已存在 | `TENANT_SLUG_EXISTS` |
| 租户不可用 | `TENANT_NOT_ACTIVE` |
| 删除确认不匹配 | `TENANT_DELETE_CONFIRMATION_MISMATCH` |
| 不能邀请超级管理员 | `INVITATION_ROLE_FORBIDDEN` |
| 最后一位管理员不可移除 | `LAST_TENANT_ADMIN` |
| 成员不属于当前租户 | `TENANT_MEMBER_NOT_FOUND` |
| 无可切换租户 | `TENANT_MEMBERSHIP_NOT_FOUND` |
| 知识库越权 | `KB_SCOPE_DENIED` |

## 11. 前端设计原则

- 平台后台、租户后台和普通用户工作台使用清晰不同的上下文标题。
- 同一上下文内菜单位置稳定，不随页面变化。
- 表格使用密排企业风格，危险操作放在行操作菜单或详情危险区。
- 删除、停用、移除必须二次确认。
- 异步按钮进入 loading 并禁用，避免重复提交。
- 成功和失败都有可见反馈。
- 表单使用真实 label、email/select 输入类型和行内校验。
- 所有图标统一使用 Lucide，不使用 emoji 充当产品图标。
- 键盘焦点可见，导航可跳过，颜色不是唯一状态表达。
- 管理表格在窄屏允许横向滚动，普通用户工作台优先适配移动端。
- 尊重 `prefers-reduced-motion`。

视觉方向延续 DocuMind 的温暖中性底色、发丝分隔和克制密度。平台后台更偏运维控制台，租户后台更偏知识资产工作台；两者共享设计语言但不共享混合导航。

## 12. 迁移策略

### 阶段 1：角色收敛

- 新增 `platform_admin`。
- 将种子超级管理员写入平台管理员表。
- 将 `enterprise_admin`、`tenant_owner` 等目标管理员迁移为 `tenant_admin`。
- 将 `user`、`analyst` 迁移为 `end_user`。
- 保留读取兼容映射，停止产生旧角色。
- 删除 `super_admin` 的知识库 ACL。

### 阶段 2：平台租户生命周期

- 实现租户创建、编辑、停用、恢复、归档和删除申请。
- 新建租户时创建首位管理员邀请。
- 系统租户列表由只读升级为可操作页面。

### 阶段 3：成员与个人中心

- 增加成员角色和状态修改。
- 增加最后一位管理员保护。
- 邀请页面支持有效期和初始权限。
- 增加个人资料、当前租户和租户切换。

### 阶段 4：导航与页面收敛

- `/system` 只保留平台菜单。
- `/admin` 只保留租户菜单。
- `/tenant` 变成兼容重定向并停止新增页面。
- 普通用户左下角统一账号菜单。

### 阶段 5：隔离加固与验收

- 增加复合外键。
- 增加双租户 API 和浏览器测试。
- 部署到 `ssh documind`，检查健康、日志和真实页面。

## 13. 验收标准

### 超级管理员

- 登录只看到平台菜单。
- 可以创建 `pending` 租户并获得首位管理员邀请链接。
- 可以设置邀请有效期。
- 可以停用、恢复、归档和发起删除租户。
- 不能默认看到租户知识库、文档和问答入口。

### 租户管理员

- 登录只看到当前租户后台。
- 访问 `/api/system/*` 返回 403。
- 可以邀请租户管理员和普通用户。
- 可以设置邀请有效期。
- 可以修改成员角色和状态。
- 不能移除最后一位租户管理员。
- 可以完整管理知识库、文档、解析、切片和 ACL。

### 普通用户

- 登录进入 `/chat`。
- 左下角可以进入个人资料、查看租户和退出。
- 访问 `/admin/*`、`/system/*` 返回 403。
- 只能看到授权知识库。

### 双租户隔离

至少创建 A、B 两个租户并验证：

- A 用户看不到 B 的知识库列表。
- A 用户不能读取、预览或下载 B 文档。
- A 的检索和引用不出现 B 的 chunk。
- A 不能获取 B 对话和问答日志。
- 缓存命中不能跨租户。
- 任务和审计均保留正确 tenant id。

### 发布验收

- 本地 Rust 和 Next.js 静态检查、测试、构建通过。
- `make deploy` 成功。
- `make status`、`make health` 和日志检查通过。
- 三角色真实登录通过。
- 双租户 API smoke 通过。
- 浏览器关键路径和截图通过。
