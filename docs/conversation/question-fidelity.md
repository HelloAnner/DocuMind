# 问题保真与灵活性 (Question Fidelity)

DocuMind 需要在两个目标之间保持平衡：一方面尊重用户真实提问，不替用户编造需求；另一方面又要支持自然追问、口语化表达、企业黑话和复杂问题拆解。

## 什么是“真实的提问”

真实提问指用户实际表达的意图、约束和上下文，而不是系统为了提高召回率自行扩展出来的问题。

示例：

| 用户原话 | 允许改写 | 不允许改写 |
|---|---|---|
| “那份合同违约责任是什么？” | “上一轮提到的 Q3 采购合同中的违约责任条款是什么？” | “Q3 采购合同中供应商延期交付的违约金比例是多少？” |
| “对比 A 和 B 的付款节点” | “对比文档 A 与文档 B 中的付款节点” | “判断 A 和 B 哪个更适合采购” |
| “有没有提到续约？” | “检索续约、自动续期、合同延期相关条款” | “总结完整合同生命周期” |

## 保真原则

- 原始问题必须完整保存，不被改写覆盖
- 改写结果必须能从原始问题和历史消息中解释出来
- 不确定指代时先澄清，不猜测
- 检索可以扩展同义词，但答案不能扩展事实
- Multi-Query 子问题必须逐条映射回用户原始问题

## 灵活性来自哪里

灵活性不来自“让 LLM 自由发挥”，而来自可控策略。

| 灵活点 | 控制方式 |
|---|---|
| 企业术语 / 黑话 | 术语表、同义词表、知识库配置 |
| 多轮追问 | 最近历史 + 指代消解 |
| 复杂问题 | Retrieval Planning 拆子查询 |
| 输出风格 | 租户级 Policy Prompt |
| 检索策略 | Top-K、阈值、HyDE、Multi-Query 开关 |
| 无答案处理 | 固定拒答模板 + 缺失证据说明 |

## 改写审计

每次改写都保存差异，便于判断系统是否偏离用户问题。

```json
{
  "original_query": "那它什么时候付款？",
  "rewritten_query": "Q3采购合同约定的付款节点是什么？",
  "rewrite_reason": "resolved pronoun from previous assistant citation",
  "added_constraints": [],
  "removed_constraints": [],
  "needs_clarification": false
}
```

如果出现 `added_constraints`，需要判断这些约束是否来自历史。没有来源的新增约束应被拒绝。

## 澄清优先级

以下情况优先澄清：

- 指代对象有多个候选文档
- 用户问“这个”“那个”“上面”但上一轮没有明确引用
- 用户要求比较，但没有说明比较对象
- 用户要求“最新”，但知识库中存在多个版本
- 用户指定“公司政策”，但可访问知识库覆盖多个制度库

## 保真校验

生成答案前进行轻量校验：

```text
original_query + rewritten_query + resolved_refs
  │
  ▼
检查 rewritten_query 是否新增实体、时间、数值、判断标准
  │
  ├── 通过 -> 进入检索
  └── 不通过 -> 回退原始问题或触发澄清
```

## 用户可见性

默认不把完整改写过程展示给用户，避免干扰问答体验。但可以在调试模式或管理员问答日志中展示：

- 原始问题
- 改写问题
- 检索关键词
- 命中文档
- 无答案原因

## 评估指标

- `rewrite.clarification_rate`：澄清率过低可能说明系统在猜
- `rewrite.added_constraint_rate`：新增约束比例
- `answer.no_source_claim_rate`：无引用支撑的结论比例
- `feedback.wrong_intent_rate`：用户反馈“答非所问”的比例
- `multi_turn.reference_resolution_accuracy`：指代消解准确率
